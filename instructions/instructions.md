# Instructions

These instructions describe the process of generating a MUS serializer for a 
user-provided defined type, such as:

```go
type Foo struct {
	num int
	str string
}

type Bar []int // []int is underlying type of Bar
```

The generated code MUST be placed in a `mus.ai.gen.go` file within the same 
package as the type. User can manually change this location or override it using 
the `// mus:path <path>` hint. 

For each `mus.ai.gen.go` generate a corresponding `mus.ai.gen_test.go` file in
the same directory.

> [!IMPORTANT]
> **1:1 Mapping**: Every single `mus.ai.gen.go` file MUST have a corresponding 
> `mus.ai.gen_test.go` file with unit tests, regardless of whether it's in a 
> subpackage, external package, or a cross-package location.

Both generated files MUST start with the following comment:

```go
// Code generated by AI agent with mus-skill. DO NOT EDIT.
```

> [!NOTE]
> **Unexported Fields**: Since the generated code is placed in the same package 
> as the user-provided type, it can access and serialize unexported fields.

## Generation Workflow

1. **Filter Generics**: If the type is a generic definition (has unspecified 
   type parameters), ignore it entirely and do not generate any code for it.
2. **Validate Type**: If the type is not a defined type, output: "Type 'type_name' 
   is not supported."
3. **Scan User Hints**: Look for `// mus:` hints. All available hints are 
   documented in [User Hints](./user_hints/user_hints.md).
4. **Interface Handling**: If the type is an interface, follow the 
   [Interface Serializer](./interface_ser/interface_ser.md) workflow.
5. **Resolve Serializers**: For each struct field or underlying type, use a 
   predefined, already generated, or anonymous serializer.
6. **Generate Tests**: Follow [Testing](./testing.md) to generate unit tests.

Follow [Multi-Package Support](./multi_package.md), [Cross-Module Generation](./cross-module.md) if needed.

## Struct Embedding

When a struct embeds another type, marshal and unmarshal it like a regular 
field using the embedded type name. For example, if `Foo` embeds `Bar`, use 
`v.Bar` to access it. 

> [!TIP]
> **Unmarshal and Skip Optimization**: For the first field of a struct, assign 
> the number of read bytes directly to `n`. For all subsequent fields, use an 
> intermediate `n1` variable and add it to `n` (`n += n1`).

## Serializer Determination Logic

To select the correct serializer for a struct field or an underlying type, 
follow these rules in order:

1. **Check Hints**: Look for an `numEnc` hint in [User Hints](./user_hints/user_hints.md). If 
   present, use the specified encoder.
2. **Use Type Mapping**: If no hint is present, use [Type Mapping](./type_mapping.md) 
   to find the default serializer.
3. **Use Serialization Mode**: Use the selected [Serialization Mode](./modes/modes.md) 
   to correct the package.

If a serializer is still not found:
1. Check if it's an anonymous type: [Anonymous Serializers](./anonymous_ser/anonymous_ser.md).
2. If it already has a generated serializer, use it. Otherwise, generate one.

The serializer must be a generated serializer or one from the 
[Predefined Serializers](./predefined_sers.md).

If still without a serializer, output: "Not supported 'type name'.".
